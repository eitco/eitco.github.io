C# SDK for .NET applications

The C# SDK provides a typed client API for arveo targeting .NET (net8.0). Like the other client SDKs, .NET applications should not call the REST API of arveo directly but use the client classes provided by this SDK.

All examples in this guide are taken from, or directly derived from, the C# integration test suite under test/system-test/src/test/csharp so that they stay aligned with the actual API.

Anatomy of the SDK

The SDK is shipped as the NuGet package Eitco.Ecr.Sdk and is made up of two complementary parts:

Generated resource clients

The bulk of the API surface is generated from the service descriptions. For every service of arveo there is a resource interface in the namespace de.eitco.ecr.common.http — for example DocumentResource, ContainerDataResource, FolderDataResource, RelationResource, MetaDataResource, TypeDefinitionResource, SearchResource and BatchOperationResource. Each interface declares the operations as default methods and provides a nested Implementation class that performs the actual HTTP calls. The generated model and identifier classes (such as Document, Container, DocumentId or TypeDefinitionName) live in sibling namespaces like de.eitco.ecr.common.document and de.eitco.ecr.common.identifier.

Hand-written convenience API

On top of the generated clients, the SDK adds manually written classes under sdk/ — most notably the fluent batch builder API in de.eitco.ecr.sdk.batch.builder (BatchBuilder, BatchCreationBuilder, BatchUpdateBuilder, …) and the EQL condition builder (EcrConditionBuilder). It also contributes a small set of partial query-element classes in de.eitco.ecr.common.search.eql (such as VariableEquals) that integrate the arveo specifics into the EQL query language.

The low-level HTTP plumbing — the HttpExecutor type used throughout the generated code — comes from the Eitco.Asdl.Runtime package, which is referenced as a dependency of the SDK.

Authentication and the HttpExecutor

Every resource client is built around an HttpExecutor, which encapsulates the target URL, the HTTP request parameters and the headers sent with each call. To authenticate against arveo, an OAuth2 access token is obtained from the identity provider (Keycloak in the test environment) and added as a bearer token header to the executor.

The test suite uses the IdentityModel library to obtain a token via the resource-owner-password flow:

Obtaining an access token (resource-owner-password flow)
HttpClient tokenClient = new();
Task<DiscoveryDocumentResponse> discoveryDocumentTask =
    tokenClient.GetDiscoveryDocumentAsync(issuerUri); (1)

discoveryDocumentTask.Wait();

DiscoveryDocumentResponse discoveryDocument = discoveryDocumentTask.Result;

if (discoveryDocument.IsError)
{
    throw new InvalidOperationException($"Error while discovering endpoint: {discoveryDocument.Error}");
}


Task<TokenResponse> tokenResponseTask = tokenClient.RequestPasswordTokenAsync(new PasswordTokenRequest (2)
{
    Address = discoveryDocument.TokenEndpoint,
    ClientId = clientId,
    ClientSecret = clientSecret,
    UserName = username,
    Password = password,
    Scope = string.Join(" ", scopes)
});

tokenResponseTask.Wait();

TokenResponse tokenResponse = tokenResponseTask.Result;

if (tokenResponse.IsError)
{
    throw new InvalidOperationException($"Error while requesting token: {tokenResponse.Error}");
}

return Token = tokenResponse.AccessToken!; (3)
1 The well-known discovery endpoint of the IDP is queried to resolve the token endpoint.
2 The token endpoint is called with the user credentials and the configured client.
3 The resulting access token is returned (and cached for subsequent calls).

The token is then attached to an HttpExecutor that points at the arveo service URL. The WithHeaderParameter method returns a new executor with the additional header, following the immutable builder style used throughout the SDK:

Creating an authenticated HttpExecutor
Authenticator authenticator = new(
    log: null!,
    clientId: TestConfig.KeycloakClientId,
    clientSecret: TestConfig.KeycloakClientSecret,
    issuerUri: TestConfig.KeycloakIssuerUri,
    username: TestConfig.KeycloakUsername,
    password: TestConfig.KeycloakPassword,
    scopes: ["openid"]
); (1)

string token = authenticator.GetToken(); (2)

Executor = new HttpExecutor(
    TestConfig.EcrServiceUrl,
    new Dictionary<string, string>(),
    []
).WithHeaderParameter("Authorization", $"Bearer {token}"); (3)
1 Configure the OAuth2 authenticator with the client and user credentials.
2 Obtain the access token (see above).
3 Build the executor for the service URL and attach the token as a bearer header.

This single executor can be shared by all resource clients — it carries the authentication and the base URL, and each resource client appends its own service path (for example api/documents or api/containers) on top of it.

Obtaining resource clients

A resource client is created from an HttpExecutor by instantiating the nested Implementation class of the corresponding resource interface. Assigning it to the interface type keeps the application code decoupled from the implementation:

Creating resource clients from an executor
DocumentResource documents = new DocumentResource.Implementation(executor);
ContainerDataResource containers = new ContainerDataResource.Implementation(executor);
TypeDefinitionResource typeDefinitions = new TypeDefinitionResource.Implementation(executor);

Each Implementation also offers constructors that take a target URL string or an existing HttpClient directly, which is convenient when no shared executor is needed:

Alternative construction
DocumentResource documents = new DocumentResource.Implementation("https://arveo.example.com");

Alternatively, the generated ApiEntryPoint class exposes factory methods (NewDocumentResource(), NewContainerDataResource(), NewSearchResource(), …) that return a fluent HttpClientBuilder for configuring the client target.

Selecting a type definition

Almost every operation is scoped to a type definition, which is identified by its name. The TypeDefinitionName class carries the type’s Id:

TypeDefinitionName typeDefinition = new()
{
    Id = "simple_invoice"
};

The available type definitions of a repository can be listed with the TypeDefinitionResource:

Listing all type definitions
TypeDefinitionResource client = new TypeDefinitionResource.Implementation(fixture.Executor);

Dictionary<TypeDefinitionId, TypeDefinition?>? result = client.GetAllDefinitions();

foreach (var (id, typeDef) in result)
{
    testOutputHelper.WriteLine($"id={id?.Identifier}, alias={typeDef?.Alias?.Id}");
}

Working with containers

Containers are metadata-only entities. They are created with the ContainerDataResource by passing the type definition and a CreateContainersInput that holds one ContainerInput per container. Attributes are supplied as a Dictionary<string, object> keyed by the field name:

Creating a container
ContainerDataResource client = new ContainerDataResource.Implementation(fixture.Executor); (1)

int integerField = 1;
long longField = 2000000000000000;
double doubleField = 3.141592653589793;
string stringField = "string";
bool booleanField = true;
TimeOnly timeField = new(12, 30, 0);
DateOnly dateField = new(2023, 1, 1);
DateTimeOffset dateTimeField = new(2023, 1, 1, 12, 30, 0, TimeSpan.Zero);
List<int> integerListField = new List<int> { 1, 2, 3 };
List<string> stringListField = new List<string> { "one", "two", "three" };

TypeDefinitionName typeDefinition = new()
{
    Id = "field_type_container"
}; (2)

List<Container?>? containers = client.NewContainers(typeDefinition, new CreateContainersInput
    {
        Inputs =
        [
            new ContainerInput
            {
                Attributes = new Dictionary<string, object> (3)
                {
                    ["integer_field"] = integerField,
                    ["long_field"] = longField,
                    ["double_field"] = doubleField,
                    ["string_field"] = stringField,
                    ["boolean_field"] = booleanField,
                    ["time_field"] = timeField,
                    ["date_field"] = dateField,
                    ["date_time_field"] = dateTimeField,
                    ["int_list"] = integerListField,
                    ["string_list"] = stringListField
                }
            }
        ]
    }
);
1 Create the container client from the shared executor.
2 Select the type definition the new container belongs to.
3 Supply the field values as a Dictionary<string, object> keyed by field name.

The created entity is returned with its server-assigned identity. The most important members of an entity are:

  • Identifier — the typed ID of the entity (for example ContainerId or DocumentId).

  • Attributes — the Dictionary<string, object?> of field values.

  • VersionInformation and ModificationInformation — version and audit metadata.

A container can be loaded again by its identifier:

Container? loaded = client.Read(typeDefinition, container.Identifier);

Working with documents

Documents extend the attribute model of containers with binary content. They are created with the DocumentResource.NewDocuments method. In addition to its attributes, a DocumentInput carries a Content dictionary that maps content element names to ContentUpload instances. A ContentUpload provides the file name and the binary payload as a Stream:

Creating the document client
DocumentResource client = new DocumentResource.Implementation(fixture.Executor) as DocumentResource;

The attributes and one content element (here a small in-memory byte[]) are passed to NewDocuments:

Uploading a document with content
List<Document?>? result = client.NewDocuments(typeDefinition, new CreateDocumentsInput
{
    Inputs =
    [
        new DocumentInput
        {
            Attributes = new Dictionary<string, object>() (1)
            {
                ["invoice_number"] = invoiceNumber,
                ["amount"] = amount
            },
            Content = new Dictionary<string, ContentUpload>() (2)
            {
                ["content"] = new() (3)
                {
                    Filename = filename,
                    Data = new MemoryStream(content)
                }
            }
        }
    ]
});
1 The attributes of the document, keyed by field name.
2 The content elements, keyed by the content element name declared in the type definition.
3 Each ContentUpload provides a file name and the binary payload as a Stream.

The returned Document exposes the uploaded content through its ContentInformation dictionary, where each entry carries metadata such as FileName and Size:

ContentInformation? info = document.ContentInformation["content"];
Console.WriteLine($"{info.FileName} ({info.Size} bytes)");

Reading documents and content

A document is loaded by its identifier with Read:

document = client.Read(typeDefinition, document.Identifier);

The binary content of a content element is streamed back with ReadContent, identified by the content element name:

Stream? data = client.ReadContent(typeDefinition, document.Identifier, "content");

The DocumentResource provides a rich set of related operations, including:

  • ReadContentRanged / ReadContentOfVersionRanged — partial (range) downloads.

  • ReadContentAs — request a specific content type, falling back to a rendition if available.

  • ReadThumbnail / ReadThumbnailOfVersion — thumbnail renditions.

  • GetVersions, ReadVersion, ImportDocumentWithHistory — version handling.

  • MoveDocumentToFolder, RemoveDocumentFromFolder, GetParentFolder — folder placement.

  • GetAnnotations, UpdateAnnotations — annotation management.

For large files, prefer the resumable TUS upload service described in Uploading data over a single NewDocuments call.

Searching for entities

Searches are issued through the data resource of the corresponding entity type (ContainerDataResource.Find, DocumentResource.Find, …) or through the SearchResource for generic and cross-type searches. They accept a SearchRequest and return a Page<T> of results.

The SearchResource offers, among others:

  • SearchGeneric / SearchEntities — search entities across types.

  • SearchChildDocuments / SearchDescendantDocuments — documents within a folder hierarchy.

  • SearchChildFolders / SearchDescendantFolders — folders within a folder hierarchy.

  • SearchRelated — entities reachable through relations.

The filter condition of a search is an EQL expression. The SDK ships a fluent EcrConditionBuilder<T> that wraps the EQL Eql.Where(…​) builder, and contributes arveo-specific query elements (such as VariableEquals, VariableIsDefined and VariableIsUndefined) for filtering on entity variables. For details on the query language itself, see Search language and the EQL documentation.

Batch operations

The hand-written batch builder API allows several create, read, update, overwrite, delete and precondition operations to be composed and committed as a single atomic transaction. It is built on the generated BatchOperationResource and is the recommended way to perform multiple operations at once.

A batch is started from the BatchBuilder, operations are appended fluently, and Commit() executes them and returns the IDs of the affected entities. Within a batch, an entity that has not been persisted yet can already be referenced by other operations through a virtual ID.

Composing and committing a batch
List<EcrId> ids = batchBuilder
    .Create()                                  (1)
        .Container()                           (2)
            .Of(containerType)
            // ... supply attributes ...
        .AndThen()                             (3)
    .Create()
        .Document()
            .Of(documentType)
            // ... supply attributes and content ...
        .AndThen()
    .Commit();                                 (4)
1 Begin a create operation. Create() returns an upsert builder; call OrUpdate() or OrOverride() before selecting the entity type to enable upsert semantics.
2 Select the entity type. Container(), Document(), Folder(), MetaEntity() and Relation() are available; an overload accepting a virtual ID lets later operations reference this entity.
3 AndThen() returns to the batch builder to add further operations.
4 Commit() executes the whole batch as one atomic transaction and returns the affected entity IDs.

The builder also supports:

  • Expect() — add a precondition that must hold at execution time, otherwise the batch fails.

  • Read() — load an existing entity into the batch result.

  • Update() / Overwrite() — merge values into, or fully replace, an existing entity.

  • Delete() — delete a single entity ById(…​), or perform a conditional bulk delete via From(typeDefinition).Where().Limit(n).Where(condition).

The generic batch concepts and semantics are described in more detail in batch-operations.adoc.

Error handling

The resource Implementation classes register exception factories for the common error types of arveo. When a request fails, the response body is deserialized into a CommonException (or CommonCheckedException) and thrown, so failures surface as typed .NET exceptions rather than raw HTTP errors.
